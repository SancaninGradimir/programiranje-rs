# --description--

Jeste li se ikada zapitali kako učiniti vaš program Express stabilnim i spremnim za produkciju, posebno kada je pod velikim opterećenjem ili pada?

Tamo gde ulaze zdravstvena provera i pravilno gašenje. Ova dva sistema pomažu da vašu aplikaciju održite dostupnom i osiguravaju da izađe čistom bez gubitka podataka ili oštećenja podataka.

Да анализирамо.

Prvo, šta je zdravstveni pregled čak i u Express?

Provera zdravlja je jednostavan način da se kaže da li je vaša aplikacija živa i funkcionalna.

Na primer:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Uređaji poput balansera opterećenja ili orkestratora kontejnera - fikiria Kubernetes au AWS Elastic Beanstalk - mogu pratiti ovaj put podataka i utvrditi da li je vaša aplikacija u dobrom stanju.

Ako ta ruta vrati statusni kod 200, vaša usluga se smatra zdravom. Zapamtite - HTTP statusni kod 200 znači da je sve u redu!

Ako ne odgovori, administrator može da samostalno ponovo pokrene uslugu.

Šta je ugodno gašenje (ili "gašiti sa počastju")?

Graciozno gašenje je ono što se dešava kada se vaša aplikacija sprema da se zaustavi, na primer kada se server ponovo pokreće ili kada se izvršava novo postavljanje aplikacije.

Umesto trenutnog prekida veze odmah, programu dajete vreme da završi ono što trenutno radi.

U Express, možete slušati `SIGTERM`, koji je sistemski signal poslat kada proces treba da se završi.

Evo primera:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Ovo ga obaveštava Express da prestane primanje novih konekcija i da sačeka da se postojeće završi pre odlaska.

Takođe treba da obradite `SIGINT`, signal koji se šalje kada pritisnete Ctrl+C da biste zaustavili program tokom lokalnog razvoja.

Dok `SIGTERM` obično koriste sistemi za orkestraciju kao što je Kubernetes u produkciji, `SIGINT` se odnosi na svakodnevni radni tok programera. Bez njegovog obrađivanja, pritiskom na Ctrl+C proces se odmah prekida, preskačući sve radnje čišćenja koje ste definisali.

Pošto oba signala treba da izazovu isto graciozno ponašanje zatvaranja, programeri obično uzimaju u obzir oba.

Sada, hajde da sve pomešamo zajedno

Zamislite da vaša aplikacija radi u kontejneru. Kubernetes šalje `SIGTERM` da ga zatvori za ažuriranje.

Pri gracioznom zatvaranju, vaš program prestaje da prima nove veze i završava sve postojeće sesije. Nakon što se svi zadaci završe, odkopcuje vezu sa bazom podataka i sigurno se izlazi. Podaci nisu izgubljeni, niti je stanje oštećeno.

Ali postoji problem — balanseri opterećenja ne znaju da se program gasi, pa mogu nastaviti da šalju novi saobraćaj ka njemu. Tu stupa na scenu dodatni sistem prikazan ispod.

Sada, imam mali poklon za tebe. Pogledaćemo "Markiranje programa jer nije u dobrom stanju pre zatvaranja".

Pre gašenja, možete ispraviti proveru zdravlja da vrati status kod 503. Na taj način, balanseri opterećenja vide odgovor o lošem stanju i prestaju da usmeravaju novi saobraćaj ka vašoj aplikaciji tokom prozora zatvaranja, kao što je prikazano u kodu iznad.

Napomena: Status kod `503` znači "Usluga nije dostupna". To ukazuje da server trenutno ne može obraditi zahteve, što se dešava posebno tokom gracioznog gašenja.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
