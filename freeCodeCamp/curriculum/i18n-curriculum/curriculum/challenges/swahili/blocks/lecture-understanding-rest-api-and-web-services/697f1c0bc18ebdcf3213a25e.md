# --description--

**REST** je skraćenica za Representational State Transfer. Zamislite ga kao set smernica koje određuju kako klijenti mreže i serveri treba da komuniciraju međusobno preko mreže.

U ovom kursu, ćeš naučiti šta je REST, njegove ključne koncepte i kako funkcioniše.

REST objašnjava kako različiti sistemi komuniciraju koristeći uobičajene mrežne tehnologije. Korišćenjem REST, klijenti kao web pretraživači, mobilne aplikacije i API komuniciraju sa serverom koristeći uobičajene metode za HTTP kao što su `GET`, `POST`, `PUT`, `PATCH` i `DELETE`, koje smo pomenuli u prošloj lekciji.

Neki od glavnih koncepata REST uključuju **bez stanja (statelessness)**, **resursi**, **načini za HTTP**, i **kod odgovora ili status**.

**Stateless** znači da svaki zahtev koji klijent pošalje serveru obrađuje se nezavisno. Server ne pamti prethodne zahteve. Zbog toga, svaki zahtev mora da sadrži sve potrebne informacije kako bi bio obrađen. Na primer, ako je korisnik autentifikovan, njegov token za autentikaciju identiteta mora biti poslat sa svakim zahtevom.

U REST, sve se smatra resursom. Resurs je deo podataka koji se može pristupiti ili modifikovati. Svaki resurs je identifikovan sa URL, kao `/products` za sve proizvode, ili `/products/1` za specifične proizvode.

Ispod je pregled metoda za HTTP, statusnih kodova i korak-po-korak primera kako arhitektura REST funkcioniše.

## Putovi za HTTP

Putanje za HTTP opisuju akciju izvršenu na resursu:

* `GET`: Pronađi resurs
    
* `POST`: Kreiraj novi resurs
    
* `PUT / PATCH`: Ažuriraj postojeći resurs
    
* `DELETE`: Obriši resurs

## Kodovi stanja za HTTP

Nakon obrade zahteva, server vraća **kod stanja** koji pokazuje rezultat:

* `200` **(OK)**: Zahtev je uspešan
    
* `201` **(Created)**: Krešiv resurs je kreiran
    
* `400` **(Bad Request)**: Klijent je poslao nevažeće podatke
    
* `404` **(Not Found)**: Zahtevani resurs nije pronađen
    
* `500` **(Internal Server Error)**: Došlo je do problema na serveru

## Kako REST radi: Korak po korak primer

### 1\. Klijent podnosi zahtev

* Klijent koristi zahtev za HTTP na način kao `GET` ili `POST`
    
* Zahtev uključuje URL koji identifikuje resurs
    
* Takođe može da uključuje zaglavlja, metadata i telo zahteva

### 2\. Server prima zahtev

* Servis validira strukturu zahteva i verifikaciju identiteta (ako je potrebno)
    
* Identifikuje ciljni resurs od URL
    
* Određuje neophodnu akciju na osnovu metode HTTP
    
* Proverava dozvole i kontrolu pristupa

### 3\. Usluga implementira akciju

* `GET`: Pronađi resurse
    
* `POST`: Kreiraj novi resurs
    
* `PUT / PATCH`: Ažuriraj resurs
    
* `DELETE`: Obriši resurs
    
* Desetine drugih važnih stvari se dešava

### 4\. Server generiše odgovor

* Server bira odgovarajući statusni kod (na primer, `200`, `201`, ili `404`)
    
* Strukturira podatke odgovora, obično kao **JSON**
    
* Dodaje relevante zaglavlja kao `Content-Type` ili `Cache-Control`

### 5\. Seva šalje odgovor

* Odgovor uključuje kod statusa, zaglavlja i opcionalno telo
    
* Stanje klijenta nije sačuvano na serveru (bez stanja)

### 6\. Klijent obrađuje odgovor

* Klijent proverava status kod kako bi utvrdio uspeh ili neuspeh
    
* Обрађује sve podatke koje je poslao server
