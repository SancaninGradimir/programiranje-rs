# --description--

Razgovarali smo malo o korišćenju grana za napraviti odvojene promene kako bismo izbegli zagađivanje vaše grane `main`. Ali šta to znači, i zašto je važno?

Ako praviš izmene direktno u svojoj grani `main`, a upstream repozitorijum prihvati izmene koje nisu tvoje, dobićeš ono što se naziva „razdvojena istorija“. To znači da commit-ovi u tvojoj grani `main` više nisu isti kao commit-ovi u grani `main` upstream repozitorijuma, što znatno otežava sinhronizaciju sa upstream-om.

Dakle, recimo da si pratio dobre prakse grananja, i da ti ovo nije problem. Ako si pratio naše prethodne uputstva, trebao bi imati fork baze podataka koju si lokalno klonirao (clone), i trebao bi imati upstream remote koji ukazuje na originalnu bazu podataka.

Ako ovo nisi uradio, evo kratkog podsjetnika. Želiš da napraviš fork baze podataka, pusnećeš svoj fork unutra, a zatim koristi ovu komandu:

```sh
git remote add upstream <url>
```

Badilisha `<url>` na URL ya SSH ya hifadhidata ya asili, SI fork yako.

Pošto imaš dobro konfigurisani upstream, možeš koristiti ovu seriju komandi da povučeš najnovije promene sa upstreama:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

Šta rade ove komande? Već si naučio kako funkcioniše `git checkout`, tako da prva komanda osigurava da se nalaziš na svojoj lokalnoj grani `main`.

Druga komanda, `git fetch upstream`, govori da se Git ide i preuzme najnovije promene sa tvog upstream remota (koji je izvorna baza podataka). Međutim, za razliku od pull-a, Git neće direktno ažurirati tvoju granu. Ovo je važno, jer je tvoja lokalna grana `main` povezana sa granom `main` na tvom forku – a ne grana `main` u izvornoj bazi podataka! Korišćenje git pull-a može dovesti do nestabilnih rezultata.

Poslednja komanda, `git merge upstream/main`, govori Git-u da spoji najnovije izmene iz grane `main` na upstream udaljenom repozitorijumu u tvoju trenutnu granu, koja je lokalna grana `main`, jer smo nju prethodno izabrali.

Ali šta ako spajanje ne uspe? Šta ako si slučajno napravio commit na grani `main` i Git ne može da spoji razdvojene istorije? Ako još nisi otvorio Pull Request iz svoje grane `main`, možeš izvršiti hard reset kako bi primorao Git da ažurira tvoju granu.

```sh
git reset --hard upstream/main
```

Komanda `git reset` omogućava ti da vratiš granu u prethodno stanje. Opcija `--hard` govori Git-u da lokalne datoteke uskladi sa stanjem grane. Time dobijaš čisto radno okruženje. Prosleđivanje `upstream/main` govori Git-u da želiš da trenutna grana (`main`) bude identična grani `main` na upstream repozitorijumu. Budi oprezan sa ovom komandom – svaki put kada prisilno menjaš Git istoriju, mogu se dogoditi neočekivane stvari.

Čim dobijete svoju lokalnu granu `main` ažuriranu odgovarajuće, važno je da potisnete taj status na vašu udaljenu *fork* (koja bi trebalo da bude `origin`, ako ste pratiti naše uputstvo). Da biste to uradili, možete koristiti:

```sh
git push
```

Ovo će gurnuti trenutno stanje na tvoj remotni fork. Ali ako si morao da uradiš hard reset kako bi očistio istoriju, moguće je da tvoj remotni fork takođe ima drugačiju istoriju i da će push propasti.

Možete popraviti to prisilnim guranjem:

```sh
git push --force
```

**Budite oprezni sa ovom komandom**. Izvršavanje primoračiće vašu udaljenu istoriju, i ako napravite grešku, primoračivanje će predstavljati veliki problem za popravku.

Guranje sinhronizovanih izmena na svoj udaljeni fork je važno jer ti omogućava da bezbedno kloniraš taj fork sa najnovijim izmenama ako izgubiš postojeći lokalni repozitorijum.

Kada vam se vaš grana `main` uskladi, možete kreirati novu granu iz tog stanja da počnete sa drugim doprinosom – sa najnovijom verzijom izvornog koda!
