## Usluga po primeru korisničke usluge

- **Server**: Računar ili program koji prima zahteve od drugog računara i šalje podatke ili usluge nazad preko mreže. Ovaj termin se odnosi na fizičku opremu, kao i na softver koji koristi.
- **Klijent-server model**: Distribuirani dizajn u kojem klijenti šalju zahteve, a server obrađuje te zahteve i vraća odgovore.
- **Klijent**: Proces ili program (poput web pregledača) koji šalje zahteve serveru.
- **Ciklus zahtev-odgovor**: Klijent šalje zahtev; server ga obrađuje i vraća odgovor.

### Vrste servera

- **Web server**: Dostavlja sadržaj mreže (HTML, slike, CSS) pretraživačima.
- **Aplikacioni server**: Upravlja poslovnom logikom i obrađuje unos korisnika.
- **Server baze podataka**: Čuva, vraća i upravlja podacima.
- **Mail server**: Šalje, prima i skladišti e-poruke.
- **File server**: Upravlja skladištenjem i distribucijom fajlova preko mreže.
- **Proxy server (ili Server proxy)**: Radi kao posrednik između klijenata i drugih servera.

## DNS (Sistem imena domena)

- **DNS**: Mrežno ime sa više nivoa i distribucija koji prevodi imena domena čitljiva za ljude u IP adrese, često nazivan "telefonbook" mreže.
- **IP Adresa**: Jedinstveni numerički identifikator uređaja na mreži (npr. `127.0.0.1` u IPv4, `0:0:0:0:0:0:0:1` u IPv6).
- **Struktura imena domena** (čitanje od desna ka levo):
  - **TLD (Najviši nivo domene)**: Najdesniji deo (npr. `.org`, `.com`).
  - **SLD (Drugi nivo domene)**: Registrovano ime (npr. `freecodecamp` u `freecodecamp.org`).
  - **Subdomena**: Rešava deo glavne domene (npr. `www`, `blog`, `api`).

### Proces rešavanja DNS

1. Pretraživač šalje ime domena **Recursive Resolver**-u (obično ga upravlja ISP).
2. **Recursive Resolver** pita **Root Servers**, koji ga usmeravaju ka odgovarajućem TLD serveru.
3. **TLD Server** vraća adresu **Authoritative Name Server** za tu domenu.
4. **Authoritative Name Server** vraća **DNS Record** sa IP adresom.
5. **Recursive Resolver** šalje IP adresu klijentu i **kešira** taj zapis kako bi ubrzao buduće pretrage.

## TCP/IP

- **Protokol**: Skup pravila i standarda koji omogućavaju uređajima komunikaciju preko mreže.
- **TCP/IP**: Kolekcija protokola koja određuje kako se podaci pakuju, šalju, rutiraju i primaju.
  - **IP (Internet Protocol)**: Usmerava pakete podataka od izvora do krajnje destinacije koristeći IP adrese. Ne zavisi od veze — svaki paket putuje slobodno bez garancije dolaska.
  - **TCP (Transmission Control Protocol)**: Razdvaja podatke u numerisovane segmente, ponovo ih sastavlja na destinaciji, proverava greške i izgubljene pakete, te traži retransmisiju ako je potrebno. Osigurava pouzdan i redosledan prenos.

## HTTP (Protokol za prenos hiperteksta)

- **HTTP**: Protokol na nivou aplikacije koji definiše pravila komunikacije između klijenata i servera u mreži.

- **HTTPS**: Sigurna verzija za HTTP gde su podaci enkriptovani korišćenjem SSL/TLS pre slanja.

### Struktura zahteva za HTTP

- **Linija zahteva**: `METHOD /path HTTP/version` (na primer, `GET / HTTP/1.1`)
- **Zaglavlja**: Parovi metapodataka ključ-vrednost (na primer, `Content-Type: application/json`).
- **Telo** (opciono): Podaci poslati serveru, koristi se sa `POST`, `PUT`, i `PATCH`.

**Standardni načini za HTTP:**

| Metoda | Svrha |
| --- | --- |
| `GET` | Dobijanje podataka |
| `POST` | Predaja novih podataka |
| `PUT` | Izmena resursa |
| `PATCH` | Ažuriranje dela resursa |
| `DELETE` | Brisanje resursa |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Struktura odgovora za HTTP

- **Status linija**: Verzija HTTP, statusni broj i opis razloga (npr., `HTTP/1.1 200 OK`).
- **Zaglavlje**: Metadata o odgovoru (npr., `Content-Type: text/html`).
- **Telo** (opciono): Vraćeni sadržaj ili opis greške.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## Brojevi stanja za HTTP

Brojevi stanja su podeljeni prema njihovoj prvoj cifri:

- **1xx** — Informativan odgovor.
- **2xx** — Uspešan odgovor.
- **3xx** — Poruka o preusmeravanju.
- **4xx** — Odgovor o grešci klijenta.
- **5xx** — Odgovor o grešci servera.

### Normalni brojevi

| Broj | Naziv | Značenje |
| --- | --- | --- |
| `200` | OK | Zahtev je uspešan; sadržaj je vraćen. |
| `201` | Created | Resurs je uspešno kreiran. |
| `204` | No Content | Uspešno, ali nema sadržaj za vraćanje. |
| `301` | Moved Permanently | Resurs je trajno premešten na novi URL. |
| `302` | Found | Resurs je privremeno premešten na drugi URL. |
| `304` | Not Modified | Resurs se nije promenio od poslednjeg zahteva; koristite sačuvanu verziju. |
| `400` | Bad Request | Server nije mogao da razume zahtev (neispravna sintaksa ili podaci). |
| `401` | Unauthorized | Zahtevana je verifikacija identiteta. |
| `403` | Forbidden | Klijent nema dozvolu za resurs, čak i sa validnom autentifikacijom. |
| `404` | Not Found | Resurs ne postoji na serveru. |
| `500` | Internal Server Error | Neočekivana greška na serveru. |
| `502` | Bad Gateway | Server je primio neispravan odgovor od gornjeg servera. |
| `503` | Service Unavailable | Server trenutno ne može da obradi zahtev. |

## Primer zahtev-odgovora za HTTP

Ciklus zahtev-odgovor sastoji se od ovih komponenti koje rade zajedno:

- **Klijent**: Pretraživač, mobilna aplikacija, ili klijent za API koji šalje zahteve za HTTP.
- **Metoda HTTP**: Akcija izvršavanja (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: Adresa zahtevanog resursa.
- **Zaglavlja**: Metadata za zahtev ili odgovor (npr., `Authorization`, `Content-Type`).
- **Telo**: Podaci poslati sa zahtevima `POST`, `PUT`, `PATCH`, ili `DELETE` (često JSON ili podaci forme).
- **Server**: Backend sistem koji obrađuje zahteve i vraća odgovore.
- **Status kod**: Broj koji ukazuje na uspeh, grešku ili preusmeravanje.
- **Telo odgovora**: Stvarni podaci koji su vraćeni, obično HTML, JSON, ili XML.

### Šest koraka ciklusa

1. **Klijent inicira zahtev** sa metodom, URL, headerima i opcionalnim telom.
2. **Zahtev putuje ka serveru** preko mreže.
3. **Server obrađuje zahtev**: proverava metodu, URL, autentifikaciju i traži bazu podataka ako je potrebno.
4. **Server priprema odgovor** sa kodom stanja, headerima i opcionalnim telom.
5. **Odgovor se šalje nazad** k klijentu.
6. **Klijent obrađuje odgovor**: prikazuje podatke za uspehom (2xx) ili prikazuje poruku o grešci pri neuspehu (4xx/5xx).

## Podnošenje obrazaca za HTML

- Forma prikuplja i šalje podatke korisnika koristeći funkciju sa komponentama `<form>` koje imaju atribute `<input>`.
- Svaki unos zahteva atribut/svojstvo `name` kako bi server prepoznao podatke.
- **Validacija na strani klijenta**: Koristite atribute `required` i `pattern` za trenutnu povratnu informaciju.
- **Validacija na strani servera**: Uvek osigurajte validaciju na serveru radi sigurnosti, bez obzira na provere sa strane klijenta.
- Atribut/Svojstvo `action` određuje gde poslati podatke forme.
- Atribut/Svojstvo `method` određuje kako ga poslati (`GET` ili `POST`; `PUT`, `PATCH`, `DELETE` zahtevaju alternative metode kao što je Fetch ili u API ili `method-override` unutar Express).

### Proces predaje forme

1. Korisnik popunjava i šalje obrazac.
2. Pregledač izvodi validaciju na strani klijenta; nastavlja ako je važeći.
3. Pregledač šalje podatke obrasca na URL koji se nalazi u `action` koristeći definisani `method`.
4. Server prima i obrađuje podatke (sačuvava u bazu podataka, autentifikuje korisnika, itd.).
5. Server može izvršiti dodatnu validaciju na strani servera.
6. Server vraća poruku o uspehu, grešku ili preusmerava.
7. Korisnik vidi povratnu informaciju u zavisnosti od odgovora servera.

## Primer mrežnih standarda i standardizacionih institucija

- **Veb standardi**: Pravila i specifikacije koje pregledači prate kako bi prikazivali veb stranice i omogućili funkcionalnosti veba.
- **WHATWG**: Organizacija za standardizaciju koja održava HTML i DOM kao „živi standard“ koji se redovno ažurira od strane glavnih timova koji razvijaju pregledače.
- **W3C**: Glavna organizacija za standardizaciju koja održava CSS, SVG, smernice za pristupačnost (WCAG) i mnoge veb API-je.
- **ECMA / TC39**: ECMA objavljuje specifikaciju ECMAScript-a (JavaScript standard), dok TC39 razvija nove funkcionalnosti jezika kroz proces od pet faza (faze 0–4).
- **Khronos Group**: Inahusika na WebGL na WebGL 2, zinazowezesha uonyeshaji wa 3D ulioimarishwa na GPU kwenye kivinjari.

## Proces kreiranja mrežnih standarda

1. **Identifikacija potrebe**: Razvojnik ili inženjer pregledača piše stvarni izazov koji je široko razmenjen.
2. **Predlog i preliminarna diskusija**: Ideja se piše neformalno kako bi se prikupili komentari i procenila zainteresovanost proizvođača pregledača.
3. **Pisanje specifikacija**: Formalni dokumenti koji opisuju sve *corner case*-ove (rubne slučajeve), stanja greške i sigurnosne implikacije.
4. **Revizija, prigovori i izmene**: Nacrt se javno pregleda; proizvođači podnose prigovore i detalji se menjaju dok se ne postigne konsenzus.
5. **Implementacija i testiranje**: Proizvođači pregledača implementiraju funkciju iza *flag*-a (indikatora); zajednički set testova osigurava doslednu implementaciju.
6. **Završetak**: Kada nekoliko nezavisnih implementacija prođe testiranje, standard je završen (W3C Recommendations, TC39 Stage 4, ili WHATWG Living Standard update).

## Životni ciklus elemenata mrežnih standarda

1. **Predlog**: Osoba identifikuje problem i piše ga kako bi saznala da li se drugi slažu.
2. **Razvoj**: Ideja se testira kroz otvorenu diskusiju; mnogo predloga umire ovde ako su retki ili nema rešenja koje zadovoljava sve.
3. **Specifikacija**: Pišu se formalne specifikacije koje pretvaraju nejasne koncepte u potpune, implementabilne pravila.
4. **Testiranje kompatibilnosti**: Set testova W3C Web Platform Tests osigurava da svaki pretraživač prepoznaje funkciju na isti način.
5. **Završen**: Mnoge uspešne implementacije znače da je funkcija dostavljena korisnicima kao osnovni deo mrežne platforme.
6. **Depreciirano**: Funkcije koje su modifikovane ili problematične označene su kao deprecirane; uklanjanje iz pretraživača je sporo zbog obaveze održavanja kompatibilnosti sa prethodnim verzijama.

## Glavni principi mrežnih standarda

- **Transparentnost**: Standardi se kreiraju otvoreno i implementiraju bez plaćanja od strane bilo koga.
- **Pristupačnost**: Mreža je dizajnirana da bude korišćena od strane svih, a pristupačnost je uključena u HTML, CSS i funkcije pregledača.
- **Kompatibilnost**: Stranica napravljena po standardu treba da radi jednako dobro na svim pregledačima i operativnim sistemima.
- **Nakon-kompatibilnost**: Nove mogućnosti se dodaju na postojeće kako bi stari sadržaj radio u savremenim pregledačima.
- **Privatnost i bezbednost**: Novi API koji mogu eksponirati podatke korisnika pažljivo se testiraju; platforma štiti korisnike po podrazumevanju.
- **Slojevanje**: HTML obrađuje strukturu, CSS obrađuje prezentaciju, a JavaScript obrađuje ponašanje — svaki sloj radi nezavisno.
- **Postepšanje**: Funkcije moraju prvo da rade za sve, a zatim se poboljšavaju za korisnike sa moćnijim pregledačima ili uređajima.
