# --description--

SQL injection je ranjivost sajberbezbednosti gde hakeri modifikuju ili koriste upitnike za pretragu koje veb aplikacija šalje bazi podataka. Na primer, hakeri mogu ubaciti zlonamerni kod SQL u upit za pretragom kako bi izvršili zlonamerni kod na backendu.

Ovo se može dogoditi kada korisnicima omogućite unos podataka putem formulara, polja pretrage ili parametara URL, a zatim korišćenjem tog korisničkog unosa za zamenu vrednosti ili konkatenaciju niza karaktera kako bi se kreirao konačni komandament za SQL.

Prema cilju ranjivosti, opasni kod `SQL` može izazvati ozbiljne poremećaje u bazi podataka i mrežnoj aplikaciji, kao što je zaobilaženje autentifikacije identiteta, krađa podataka iz baze podataka, ili modifikovanje/izbrisanje zapisa u bazi podataka. Stoga, trebalo je da potraktujete ovu pretnju veoma ozbiljno.

Na primer, recimo da vaša veb aplikacija koristi parametar SQL za autentifikaciju korisnika, direktno unoseći korisničko ime i lozinku koje je korisnik unio:

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

Ako ubodač unese nešto slično u deo `username`:

```sql
'" " OR "1"="1" --'
```

I u bilo kom drugom delu sekcije `password`, jer `"1"="1"` je uvek `true` i `--` dodaje komentar za preostali deo argumenta, odeljak `WHERE` postaje:

```sql
WHERE username = " " OR TRUE
```

Uvek je `true`.

U zavisnosti od implementacije baze podataka, ovo može vratiti podatke prvog korisnika ili bilo kog korisnika u bazi podataka i omogućiti ranjivosti da pristupi bez važećih sertifikata.

Pa, kako ga možete sprečiti? Postoje dva glavna načina da se blokira SQL injekcija:

- Nemoj praviti dinamičke SQL upite spajanjem stringova.
- Sprečava ubacivanje zlonamernog SQL koda u upite koji se izvršavaju na serveru.

U praksi, možete koristiti parametrične izjave, poznate i kao parametrični argumenti. Oni razdvajaju strukturu SQL od podataka koje je korisnik unio, što sprečava bazu podataka da interpretira ulazne podatke kao izvršni kod.

Drugi važan metod je validacija unosa. Preporučuje se kao dodatna bezbednosna mera u svakom slučaju, ali je posebno važno za delove argumenta SQL koji su povezani sa nazivima tabela, nazivima kolona i redosledom sortiranja. Ako je moguće, savetuje se da nazive tabela i nazive kolona unesete direktno u svoj kod, a ne iz unosa korisnika.

Ako, uprkos svim preventivnim merama, naiđete na napad tipa SQL injekcije, možete smanjiti njegove posledice na bazu podataka ograničavanjem dozvola svakog naloga baze podataka. Dajte svakom korisniku baze podataka samo dozvole koje su mu potrebne za njegov zadatak, a ne više. Ako nalog samo treba da čita podatke iz baze podataka, date mu samo dozvolu za čitanje. I kao preventivnu meru, nikada nemojte dodeljivati administratorske dozvole vašim aplikacionim nalogima. Ovo je veoma opasno jer zlonamerni kod (bug) može dobiti potpun pristup i kontrolu nad bazom podataka.

SQL injection je veoma uobičajen napad koji vi, kao programer, morate razumeti i sprečiti prilikom implementacije vaše web aplikacije. Ovim metodama, možete zaštititi svoju bazu podataka i osigurati podatke vaših korisnika.
