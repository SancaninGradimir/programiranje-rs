# --description--

Jesi li se ikada zapitalo šta se dešava kada nešto pukne u vašoj aplikaciji za Express?

Šta ako se konekcija sa bazom podataka sruši? Ili kada korisnik pristupi putanji podataka koja ne postoji?

Razumeti kako funkcioniše rukovanje greškama u Express je važno ako želite da izgradite sigurne i pouzdane web aplikacije.

U ovom tutorijalu, pregledaćemo kako Express rukuje greškama, i kako možete da ih sprečite i pružite odgovore koristeći stvarne primere koda.

## Kako rešiti jednostavnu grešku u Express

U Express, handler grešaka je izgrađen na centralnom programu.

Kao i standardne funkcije centralnog programa, handler za greške ima pristup `req`, `res`, na `next`. Ali ima jedan dodatni parametar na početku: `err`.

Ovo je sintaktička struktura:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Ovo opisuje ovaj centralni program na kraju vašeg reda centralnih programa.

Express zna da je ovo obrađivač grešaka zato što ima četiri parametra — novi `err`, zajedno sa standardnim parametrima `req`, `res` i `next` za druge middleware funkcije.

U sledećoj lekciji naučićete više o middleware-u za obradu grešaka i kako se uklapa u širi Express middleware lanac.

## Kako rešiti asinhrone greške

Rešavanje asinhronih grešaka zavisi od verzije Express koju koristite.

Zapamtite, bez obzira na verziju Express koju koristite, i dalje morate definisati svoj intermediarni program za rukovanje greškama na kraju svoje sekvence, nakon svih vaših putanja podataka.

Ovo je opet rukovodilac grešaka:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

U Express 4 dole, što i dalje koriste mnogi projekti, nestovremene greške **nisu automatski uhvaćene**.

Ukoliko koristiš `async`/`await` sa nečim koje baca unutar rukovaoca putanje podataka za `async`, Express neće uhvatiti osim ako mu ručno ne date `next()`.

Ovo je kako obrađujete greške koje nisu istovremene u Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Ako zaboravite deo `try/catch` i `next(err)`, greška može:

* Kusababisha programu yako kuanguka
* Au kusababisha onyo la ahadi isiyoshughulikiwa

Zato je važno obuhvatiti asinhronu logiku u `try`/`catch` blok u Express 4.

Neki programeri čak kreiraju funkciju za pomoć da bi izbegli ponavljanje `try/catch` svuda:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Zatim ga koristite ovako:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

Ovo vam omogućava da ne morate koristiti fragmente `try/catch` u svim vašim putevima za `async`, dok istovremeno pravilno prenosite grešku.

## Express 5

Trenutna verzija od Express je poboljšala ovu funkciju.

Od Express 5, ako je putanja podataka za `async` baci grešku ili vrati odbijeno obećanje, Express ne šalje direktno centralnom handleru za greške.

Ovo znači da ovaj primer obrađuje grešku bez `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Ako `getUserFromDatabase()` vrati grešku, Express 5 će je uhvatiti i direktno obezbediti rukovaoca greškom.

Nije potreban poziv funkcije `next(err)`.

## Kako rešiti greške 404

Greška 404 je malo drugačija. Nastaje kada ne postoji put podataka koji odgovara zahtevu.

Možete ih upravljati pomoću ove centralne aplikacije:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Važno**: Ovo treba da se postavi nakon vaših putanja podataka, ali pre centralnog programa za obradu grešaka.

Konačno, ovo su najbolje metode za rešavanje grešaka u Express:

* Uvek stavi svoj centralni program za rukovanje greškama na kraj.
* Obradi različite greške 404.
* Koristi `next(err)` za guranje grešaka iz puta ili logike `async` u Express 4 i smanjenju.

U produkciji, ne šalj niz grešaka korisnicima. Unesite detalje unutra i vratite prijateljsku poruku korisniku umesto toga.

Rukovanje greškama u Express se odnosi na implementaciju sredinskog softvera koji sluša probleme i pruža odgovore na jasan i siguran način. Bilo da je reč o grešci u vašem kodu ili nedostajućoj putanji, vaša aplikacija nikada ne bi trebalo da padne bez plana za hitne slučajeve.
