# --description--

Ako si ikada bio u društvu iskusnih programera, bilo na internetu ili uživo, verovatno si čuo za „strašne konflikte pri spajanju“. Ali šta su oni zapravo?

Kada koristiš dobru strategiju grananja, kao što smo govorili u prethodnoj lekciji, možeš istovremeno raditi na dve različite funkcionalnosti u dve različite grane. Ili možda radiš na istoj funkcionalnosti dok tvoj kolega radi na drugoj grani. U svakom slučaju, postoje dve nezavisne izmene. Međutim, ponekad će obe grane menjati istu datoteku. Ako se druga grana prva spoji u `main`, izmene u tvojoj grani će zaostajati.

To se često može rešiti ponovnim spajanjem grane `main` u tvoju granu. Međutim, kada su izmene iz druge grane u sukobu sa tvojim izmenama, Git ne može da odluči koje izmene želiš da zadržiš tokom spajanja. To se naziva „konflikt pri spajanju“.

Kako ga možeš rešiti? Za manje i jednostavnije konflikte, GitHub nudi veb interfejs za rešavanje. Dugme „Resolve conflicts“ pojaviće se ispod Pull Request-a ako postoje konflikti. Kada ga klikneš, videćeš datoteku sa konfliktima koja sadrži tvoj kod i kod koji je u sukobu. Git će dodati „oznake konflikta pri spajanju“ oko oba dela koda kako bi pomogao da se problem identifikuje. Izgleda ovako:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

Otvarajući marker je `<<<<<<< feat/conflict`, i on pokazuje početak promena koje naša trenutna grana vrši. Zatim vidimo `Here's some code from my first PR :)`. Nakon toga je marker `=======` koji pokazuje kraj naših promena i početak promena za granu `main`. Možemo videti da su promene napravljene u `main` linija `Here's some code that was merged to main that conflicts with the code on my PR`. I nakon toga je zatvarajući marker `>>>>>> main` koji pokazuje kraj konflikata.

Ali, kako da to rešimo? Dobro, morate odlučiti koje promene želite da sačuvate kao deo konflikata spajanja (merge conflicts). Ako želite da sačuvate obe promene, samo obrišite markere konflikta i ostavite promene neizmenjene. Inače, ako želite da sačuvate promene samo iz jedne grane, pronađite markere konflikta i promene iz druge grane (one koje ne želite da sačuvate). I konačno, ako ne želite nijednu promenu uopšte, možete obrisati sve od početnog markera do krajnjeg markera (uključujući).

Obrišimo izmene iz grane `main`, ali zadržimo naše. Sada imamo sledeće stanje:

```sh
1 Here's some code from my first PR :)
```

Klikni na dugme „Mark as resolved“, čime GitHub-u govoriš da si zadovoljan rešenjem. Ako postoji još datoteka, GitHub će preći na sledeću. Međutim, pošto je ovo jedina datoteka sa konfliktima, videćeš dugme „Commit merge“. Klikni na njega i GitHub će napraviti merge umesto tebe. Bićeš vraćen na Pull Request gde ćeš videti novi merge commit i rešene konflikte. I to je to!

Ali šta ako konflikte ne možeš da rešiš direktno na GitHub-u zato što su previše složeni?

Ako tvoja grana ima mali broj commit-ova ili samo nekoliko njih menja tu datoteku, možda možeš spojiti `main` u svoju granu:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Kada Git izvršava ovo spajanje, dobićeš konflikte koje moraš ručno da rešiš. Git će ti pokazati koje datoteke zahtevaju ispravke. Kada otvoriš datoteke sa konfliktima u editoru, videćeš iste oznake konflikta, kao i korisne dodatke korisničkog interfejsa u VS Code-u. Reši konflikte na isti način kao ranije. Ovog puta moraćeš ručno da napraviš commit. To je isto kao i svaki drugi commit:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

Nakon izvršavanja ovih komandi, možemo videti Git da je prihvatilo našu rezoluciju.

Ponekad, kada imate mnogo commit-ova ili mnogo konflikata, rebase može pomoći. Umesto spajanja promena (merging), rebase uzima sve commit-ove sa vaše grane i postavlja ih tako da budu nakon nedavnih promena na `main`. Rečima Git, ovo efektivno "resetuje" osnovu vaše grane na najnoviji commit u `main`.

Da bismo ovo testirali, nam treba nova grana:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

Hajde da počnemo kreiranjem mnogo commit-ova na našoj grani, kako bismo mogli da vežbamo rebase:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

Sada, napravićemo još izmena u `main` kako bismo stvorili konflikt:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

I odaberite našu granu za rebase:

```sh
git checkout feat/rebase
```

Zatim pokreni `git rebase -i main`. Ovo će uraditi rebase tvoje grane na trenutno stanje grane `main`. Opcija `-i` govori Git-u da izvrši interaktivni rebase. Videćeš svoje commit-ove sa rečju `pick` pored njih. `pick` govori Git-u da koristi commit bez izmena tokom rebase-a. Sačuvaj i zatvori datoteku, a zatim ćeš dobiti poruku o grešci koja ukazuje na konflikte.

Naš commit `change one` je u sukobu sa commit-om `change four` i moramo ga rešiti. Otvori datoteku u editoru i videćeš isti interfejs za konflikte kao ranije. Zadrži naše izmene iz `change one`. Zatim pokreni `git add .` da dodaš izmene i `git rebase --continue` da kažeš Git-u da nastavi rebase. Pošto smo rešili konflikt u prvom commit-u, Git će preostale commit-ove obraditi automatski.

Sada, kao konačno čišćenje, uradimo *squash* naših tri commita promena u jedan commit za čistiju istoriju. Endesha `git rebase -i main` ponovo da dođeš do editora. Zatim, ostavi prvu komandu kao `pick`, ali promeni sledeće dve u `squash`. Možda ćete naići na isti konflikt ponovo. Ako ga naiđete, rešite ga na isti način kao što ste to prvobitno uradili. Čim ga rešite i nastavite sa *rebase*-om (ili odmah, ako nema konflikta), videćete da se otvara editor za poruku novog commita. Ova poruka će biti vaš novi commit, koji sadrži promene iz tri commita koje smo spakovali. Tuitaje ovo `change three complete`. Sačuvajte i zatvorite editor, a Git će završiti *rebase*.

Ako pregledamo tvoju istoriju commita sada, možemo videti da imamo jedan commit za `change three complete`, i da je savršeno postavljen na osnovu commita `change four`. To znači da si uspešno završio rebase i spajanje commita! I sada imaš alate koji su ti potrebni da postaneš majstor konflikata spajanja!

Nakon rešavanja problema ručnog povezivanja u vašoj lokalnoj bazi podataka, koja komanda vam je potrebna da završite povezivanje?

## --answers--

`git push origin main` na `git rebase -i origin/main`

### --feedback--

Nakon rešavanja konflikata, moraš da dodaješ i napraviš komit promena koje su rešene kao bilo koji drugi komit.

---

`git add` i `git commit`

---

`git merge --abort` i `git commit`

### --feedback--

Nakon rešavanja konflikata, potrebno je da dodate i napravite commit za rešene promene, kao što je bilo koji drugi commit.

---

`git reset --hard` na `git push -f`

### --feedback--

Nakon rešavanja konflikata, moraš da stage-uješ i napraviš commit za izmenene fajlove kao bilo koji drugi commit.
